Collated design audit — 14 suggestions

Two independent audits of the src-ui React + Tailwind v4 + Base UI front-end were run: one through the Emil design-engineering lens (micro-interaction craft, animation discipline) and one through the Apple design lens (fluid interfaces, materials, typography, wayfinding). Overlapping findings are merged, everything is re-ordered by impact, and each item carries a senior verdict. Every card demonstrates Current vs Proposed live — press, hover, and replay the demos. Mark each item Approve / Deny / Discuss, add notes, then copy the decision summary at the bottom and paste it back to Claude.

High Medium Low Source: Emil / Apple / Both
1

Press-down feedback on Button (+ replace transition-all)

High Both

No control in the app responds on pointer-down — buttons only react on hover and release, so the UI feels like it isn't listening. Both audits flag this as the single highest-leverage polish fix since it flows through every pressable. Merged in: Emil's note that the same base classes use transition-all, which animates unintended (including layout) properties.

src-ui/src/components/ui/button.tsx:8

- "… transition-all …"
+ "… transition-[transform,background-color,border-color,color,box-shadow] duration-150 ease-out active:scale-[0.97] …"
Current

Press and hold — nothing acknowledges the press until release.

Proposed

Press and hold — active:scale(0.97) at 160ms ease-out confirms the press instantly.

Agree. Foundation-level fix; one line in the shared variant covers the whole app. Ship the explicit transition list in the same change — they touch the same string.
2

No prefers-reduced-motion handling anywhere

High Both

A grep for reduced motion returns zero hits. Perpetual ping, spin decorations, and zoom/slide entrances all run for motion-sensitive users. Reduced motion means gentler, not zero: keep opacity fades and functional spinners, drop decorative loops and spatial entrances. One global CSS block covers the app.

src-ui/src/index.css (global)

@media (prefers-reduced-motion: reduce) {
  .animate-ping { animation: none; }
  [data-state] { /* entrances */ animation-duration: 1ms; }
  /* keep functional spinners; swap zoom/slide for fade-only */
}
Current
〰️ Ping loops, zoom entrances, spins — always on.
⚠️ Vestibular triggers ignored.

Static illustration — the OS setting is simply never read.

Proposed
✓ Opacity fades kept; loops and slides removed.
✓ Functional spinners stay (slower is fine).

This review page itself honors the setting — its looping demos calm down under reduced motion.

Agree. Accessibility baseline, trivially cheap. Do it as one global media block plus a check on any JS-driven animation added later.
3

Connection dot pings forever

Medium Both

The connected indicator runs animate-ping infinitely. Both audits independently flag it: a permanent status element looping forever is an attention leak (Emil) and a vestibular trigger Apple explicitly warns against (Apple). Animate the transition to connected — 2–3 pings — then settle to a static dot.

src-ui/src/components/connection-status.tsx:50

- <span className="animate-ping …" />
+ <span className="animate-ping [animation-iteration-count:3] [animation-fill-mode:forwards] …" />
  (re-key the element on connect so the 3 pings replay per event)
Current
Connected — pings forever

Your eye keeps getting pulled here, indefinitely.

Proposed
Connected — pings 3×, then rests

animation-iteration-count: 3 — celebrates the change, then gets out of the way.

Agree. Textbook "animate state changes, not steady state". Trigger on the connect event, not on mount.
4

Tooltips open with zero delay

Medium Emil

TooltipProvider sets delay = 0, so every incidental hover fires a tooltip — the sidebar becomes noisy. Correct behavior is the inverse of what ships: delay the first hover (~500ms), then open instantly for adjacent targets (Base UI grouping handles the warm-up).

src-ui/src/components/ui/tooltip.tsx:6

- function TooltipProvider({ delay = 0, … })
+ function TooltipProvider({ delay = 500, … })
Current Hover meFires instantly

Just passing the pointer across triggers it.

Proposed Hover meAppears after 500ms

500ms of intent required; neighbors would then open instantly.

Agree. One-token change; verify Base UI's provider grouping keeps the instant-for-neighbors behavior after the delay bump.
5

Dialog entrance pops instead of entering

Medium Emil

Dialog and AlertDialog animate in at duration-100 with the weak default easing — it reads as a "pop", not an entrance. Modals belong in the 200–500ms band with a strong custom ease-out; exits should stay faster (~150ms). The existing zoom-in-95 + fade start values are correct and should be kept.

src-ui/src/components/ui/dialog.tsx:32,54 (and alert-dialog)

- "… duration-100 …"
+ "… duration-200 ease-[cubic-bezier(0.23,1,0.32,1)] data-[state=closed]:duration-150 …"
Current
Push anyway?Not enough free space on Pixel.

100ms, default ease — blinks into place.

Proposed
Push anyway?Not enough free space on Pixel.

200ms, cubic-bezier(0.23, 1, 0.32, 1) — decelerates into place.

Agree. Cheap and correct; keep exit faster than enter, and do not touch the (already right) scale-from-0.95 start.
6

Fake material on the title bar

Medium Apple

PageHeader applies backdrop-blur-lg over a fully opaque bg-sidebar — the blur is dead code and the chrome reads as a flat strip. On macOS, translucent chrome is the native expectation. Either make it a real material (bg-sidebar/75 + blur with content scrolling underneath) or delete the blur. If translucency is adopted, pair with a prefers-reduced-transparency solid fallback and a scroll-edge border/fade.

src-ui/src/components/header.tsx:18 · src-ui/src/components/app-page-chrome.tsx:60

- className="… bg-sidebar backdrop-blur-lg …"
+ className="… bg-sidebar/75 backdrop-blur-lg supports-[backdrop-filter]:bg-sidebar/75 …"
Current
Photo Bridge

Scroll the content — the bar is opaque; the blur does nothing.

Proposed
Photo Bridge

Scroll — 60% background + real backdrop blur; content glows through the chrome.

Agree with caveat. Material honesty is right, but verify Tauri/WKWebView renders backdrop-filter over the scrolled content correctly before committing; otherwise take the "remove the blur" branch.
7

Ad-hoc micro type scale & stacked contrast reductions

Medium Apple

Five near-identical sizes coexist: text-[10px], text-[11px], text-[0.8rem], text-xs, text-sm. 10px is below comfortable desktop legibility, and the smallest text also gets text-muted-foreground/80 — two contrast reductions stacked on the smallest type. Consolidate with 12px as the floor; build hierarchy with weight and color, not 1px deltas.

src-ui/src/routes/transfer-media.tsx:227,269,285 · src-ui/src/components/ui/button.tsx:27 · src-ui/src/components/app-sidebar.tsx:273

- <p className="text-[10px] text-muted-foreground/80">Advanced</p>
+ <p className="text-xs font-medium text-muted-foreground">Advanced</p>
Current
Advanced Skips HEIC files already converted in a previous run. Recommended for repeat transfers

Actual rendering: 10–11px at 80% muted opacity.

Proposed
Advanced Skips HEIC files already converted in a previous run. Recommended for repeat transfers

12px floor; hierarchy via weight + one muted step, never both size and opacity.

Agree. "Muted color OR small size, not both" is the rule to encode; sweep the listed call sites and add a lint-note to the design system.

Whole-app lockout & "dead" 50% sidebar during jobs

Medium Both

Every nav item disables while any operation runs, and the default disabled:opacity-50 dims the whole sidebar — it reads as "app is broken" (Emil) and traps the user entirely (Apple). Since Pixel/job context lives above the routes, navigation can likely stay open while the job persists; at minimum keep Settings reachable and style busy items with aria-disabled that keeps labels legible.

10

Resize grip loses fast drags (mouse events, no capture)

Medium Both

The activity-panel resize uses mousemove/mouseup on document: fast drags that exit the window drop the handle, and touch/pen do nothing. Pointer Events with setPointerCapture keep the drag glued to the grip regardless of pointer speed or input type. Direct 1:1 height tracking during drag is correct — keep it; just clamp min/max and persist on release.

src-ui/src/components/activity-feed.tsx:35-59

- document.addEventListener("mousemove", onMove); document.addEventListener("mouseup", onUp);
+ onPointerDown={(e) => { e.currentTarget.setPointerCapture(e.pointerId); … }}
+ onPointerMove / onPointerUp on the grip itself — no document listeners needed
Current
mousemove on document
pointer leaves window → drag lost ✗
touch / pen: no response

Static illustration — the failure needs a fast off-window drag to reproduce.

Proposed
setPointerCapture(pointerId)
pointer leaves window → drag continues ✓
touch / pen: works

Capture pins all pointer events to the grip until release.

Agree. Strictly better API for the same code shape; also removes the document-listener cleanup footgun.
11

ThemeToggle segmented control: layout animation + invisible to a11y

Medium Both

The highlight animates width/left via inline style (layout + paint every frame), measured in a useEffect keyed on theme — a window resize or font load desyncs it. All three segments are equal size-8, so a pure translateX is cheaper and can't desync. Merged in (Apple craft note): the buttons are raw <button>s with no focus-visible style and no aria-pressed/radio semantics.

src-ui/src/components/theme-toggle.tsx:40-45,84,89-105

- style={{ left: rect.left, width: rect.width }}   // measured per theme change
+ style={{ transform: `translateX(${index * 32}px)` }}  // index-derived, no measurement
+ role="radiogroup" on the wrapper, role="radio" + aria-checked per button, visible focus ring
Current

Click segments — highlight animates left/width (layout thrash; desyncs on resize in the app).

Proposed

Same look via translateX(index × 32px) — composited, index-derived, tab to see the focus ring.

Agree. Two audits hit the same component from different angles — fix performance and semantics in one pass.

13

Theme switch is an abrupt brightness jump

Low Apple

The theme provider swaps the .dark class instantly — a hard full-screen brightness cut. A ~200ms color-only transition on root tokens softens the flash. Must be guarded: apply a temporary class only during the swap so colors aren't transitioning on every hover forever.

src-ui/src/components/theme-provider.tsx

root.classList.add("theme-transition");        // .theme-transition * { transition: background-color 200ms, color 200ms, border-color 200ms }
root.classList.toggle("dark", isDark);
setTimeout(() => root.classList.remove("theme-transition"), 250);
Current
Convert MediaDrop HEIC files here to convert.

Hard cut — the whole surface flashes.

Proposed
Convert MediaDrop HEIC files here to convert.

200ms color cross-fade — same destination, no flashbulb.

Agree with caveat. Nice-to-have; the guard class is mandatory — an unguarded global color transition would poison every hover/state change in the app.
14

Low Emil

After a scan completes, the file-list day groups blink in simultaneously. A 30–50ms CSS animation-delay stagger on first render only, capped at ~6 items, is the one cheap delight moment in the app. Never on re-renders, never blocking interaction.
