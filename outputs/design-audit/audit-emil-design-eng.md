# Photo Bridge — Design Engineering Audit (emil-design-eng skill)

Scope: `src-ui` (Tauri desktop app UI — React + Tailwind v4 + Base UI + shadcn-style components).
Philosophy applied: unseen details compound; taste is the differentiator; every animation must justify itself.

| Before | After | Why |
| --- | --- | --- |
| `transition-all` in `buttonVariants` base classes (`components/ui/button.tsx:8`) | `transition-[transform,background-color,border-color,color,box-shadow] duration-150 ease-out` | Never use `transition: all` — it animates properties you didn't intend (layout-affecting ones included) and makes future regressions invisible. Specify exact properties. |
| No `:active` state on any button (`components/ui/button.tsx`) | Add `active:scale-[0.97]` to the base variant (with `transition-transform 160ms ease-out` already covered above) | Buttons must feel responsive to press. Scale-on-press (0.95–0.98) is instant confirmation the UI heard the user. This is the single highest-leverage polish fix — it applies to every pressable in the app. |
| `TooltipProvider` sets `delay = 0` (`components/ui/tooltip.tsx:6`) | `delay = 500` at the provider, keep instant open for adjacent tooltips (Base UI grouping handles this) | Instant tooltips fire on every incidental hover — the sidebar becomes noisy. Tooltips should delay on first hover, then open instantly for neighbors. Currently the app has the exact inverse of the right behavior. |
| Dialog/AlertDialog use `duration-100` with default easing (`components/ui/dialog.tsx:32,54`) | `duration-200` with a strong ease-out, e.g. `ease-[cubic-bezier(0.23,1,0.32,1)]`; keep exit at ~150ms | 100ms with a weak default curve reads as a "pop", not an entrance. Modals sit in the 200–500ms band. Built-in easings are too weak — use a custom curve. Exit should be faster than enter. |
| `zoom-in-95` + `fade-in-0` on dialog — correct starting scale ✓ | Keep — but add `transform-origin: center` awareness only for popovers/menus, not the dialog | Modals correctly stay centered; scale(0.95)+opacity is the right entry. No change needed here — noted so it isn't "fixed" wrongly. |
| Connected indicator: `animate-ping` runs forever (`components/connection-status.tsx:50`) | Static green dot, or ping 2–3 iterations after a connect event then settle (`animation-iteration-count: 3`) | An infinite looping animation on a permanent status element has no purpose after the first second — it pulls the eye away from the user's actual task, forever. Animate state *changes*, not steady state. |
| Spinners: `IconLoader2` + Tailwind `animate-spin` (1s/rev) everywhere | `[animation-duration:600ms]` on loading spinners | A faster-spinning spinner makes the same wait feel shorter. 1s/rev reads as lethargic; ~0.6s reads as "working hard". |
| ThemeToggle highlight animates `width` via inline style (`components/theme-toggle.tsx:40-45,84`) | Animate `transform: translateX(...) scaleX(...)` only, or since all three buttons are equal `size-8`, drop the width var entirely and just translate | Animating `width` triggers layout+paint every frame. All segments are the same size, so a pure translate is both cheaper and simpler. Also: measurement runs in a `useEffect` keyed on theme — window resize desyncs the highlight. |
| No `prefers-reduced-motion` handling anywhere in `src-ui` (grep: zero hits) | Global guard: `@media (prefers-reduced-motion: reduce)` — keep opacity fades, drop `animate-ping`, `animate-spin` decorations (keep functional spinners), zoom/slide entrances | Motion sensitivity is real; reduced motion means gentler, not zero. One CSS block covers the whole app. |
| Log/activity panel resize uses `mousemove` on `document` and sets `height` px per frame (`components/activity-feed.tsx:35-59`) | Use Pointer Events + `setPointerCapture`; keep height set (resize is genuinely a layout change) but clamp min/max and store on release | Mouse events don't capture — fast drags outside the window lose the handle; touch/pen do nothing. Pointer capture keeps the drag glued. (Direct 1:1 tracking during drag is correct — don't add animation here.) |
| Drag-over state on dropzone only tints colors (`components/empty-dropzone.tsx:22-25`) | Add a subtle `scale(1.01)` + stronger border on `isDragging`, `transition-[transform,background-color,border-color] 200ms ease-out` | The drop target should visibly "reach" for the payload. Color alone is easy to miss in peripheral vision; a 1% scale is felt, not seen. |
| Sidebar nav items disabled while a job runs, with tooltip (`components/app-sidebar.tsx:246-262`) | Keep the guard, but don't dim every item with `disabled:opacity-50` default — use `aria-disabled` styling that keeps labels legible | Fine to block navigation mid-job, but a whole sidebar at 50% opacity reads as "app is broken". Communicate "busy", not "dead". |
| File-list / day-group content appears all at once after scan (`routes/browse.tsx`) | Optional: 30–50ms stagger on the first render of day groups (CSS `animation-delay`, capped at ~6 items) | Occasional, post-loading moment — the one place delight is cheap. Cap the stagger so long lists don't feel slow; never block interaction. |

## Notes on what's already right

- Entry animations use `zoom-in-95` + fade — nothing animates from `scale(0)`.
- `select-none` on chrome, `select-auto` restored in log output — thoughtful.
- Loading → success/error toast morphing via `toast.loading(...)` + `{ id }` reuse is exactly the right state-indication pattern.
- Preview request-id guard (`manage-device.tsx:99-120`) invisibly handles the stale-async edge case — the Sonner principle of handling edge cases invisibly, applied well.
